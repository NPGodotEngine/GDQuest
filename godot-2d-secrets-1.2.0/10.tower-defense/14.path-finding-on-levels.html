Finding the path to the base

In the Wave System lesson, we looked at how to move Enemies on a path. We can design paths on our own using the built-in editor when designing a Level. But now that Waves spawn during run-time, we don’t have this ability anymore. On top of that, it can be a tedious task to draw the paths our own and we can’t change paths during run-time. What would happen if the player blocks the current path?

We need to automate the task of creating and calculating new paths. For that, we’ll use a TileMap and Godot’s built-in A* implementation, the AStar class.

In this section, we’ll see how to find a path between two points using the AStar algorithm. You’ll learn to:

  • Map cells and world coordinates
  • Generate a smart path with AStar algorithm

Using the AStar class

The A* (pronounced A star) is a pathfinding algorithm we often use due to its performance and accuracy. Godot Engine has a built-in implementation named AStar. Since our game is in a 2D world, we can use its variant, AStar2D.

It inherits from the Reference class so its usage is a bit different compared to Nodes and Resources. As the documentation states:

You must add points manually with add_point and create segments manually with connect_points. Then you can test if there is a path between two points with the are_points_connected function, get a path containing indices by get_id_path, or one containing actual coordinates with get_point_path.

We create an abstract world to manage these points, segments, and paths. Remember how we discussed abstracting game worlds into grids with the TowerPlacer? We’re going to do that again.

Our path finding approach uses a TileMap node to map world coordinates to a discrete space. This decreases the amount of points it handles and connects. Let’s create a new scene with a TileMap as root and rename it as AStarGrid. Attach a new script to it. Save the scene as res://Levels/AStarGrid/AStarGrid.tscn. and save the script in the same folder.

Open the AStarGrid script and create a new instance of AStar2D by calling the method new() on it. The variable _astar is a pseudo-private variable because outer classes shouldn’t work on it directly. They should only do so with the public methods we provide. This is to ensure the path the AStarGrid creates will go through all its procedures.

# Calculates a set of walkable positions between the start_point and the
# goal_point
extends TileMap

var _astar := AStar2D.new()

Creating AStar points

As the reference states, we use the add_point() method to feed the AStar2D with points. But we want the client class, in our case the Level, to provide a list of available points instead. This so our AStarGrid calculates the path itself.

We have cells which only Towers can sit, cells that are obstacles, and cells the Enemies can walk on. Our AStarGrid only cares about the cells the Enemies can walk on. Create a walkable_cells property to store cells that build up this walkable path.

We need to find a path between the start and goal point. Let’s export two Vector2 variables, the start_point and the goal_point. This will allow us to set these points ourselves if we want. But ideally, the Level will provide these points. The start_point is going to be where Enemies spawn. The goal_point is the PlayerBase’s position.

We create the AStar points inside a method so we can call it whenever we need it. Let’s name this procedure _create_astar_points().

The AStar requires that each point added through the add_point() pairs with an ID. This allows us to identify each point later on.

We use an iterator variable to index each point based on the order we add it. We create a cell_id variable then we run through each cell in the walkable_cells list. For each cell, we call the _astar.add_point() and pair it with a new cell_id.

We need to store the IDs of the start_point and goal_point as we iterate. This is because the get_point_path() method only accepts IDs as arguments. We create two member variables: _start_id and _goal_id. We need to check if the current cell matches the start or the goal cell positions as we iterate so we map the start_point and the goal_point using the world_to_map() method. If the cell matches, we store it on its respective member variable. Finally, we increment the cell_id.

export var start_point := Vector2.ZERO
export var goal_point := Vector2.ZERO

var walkable_cells : PoolVector2Array

var _start_id := 0
var _goal_id := 0

# Add all the points inside the `walkable_cells` to the AStar instance
func _create_astar_points() -> void:
    # Sets cells ID by iteration order
    var cell_id := 0
    for cell in walkable_cells:
        _astar.add_point(cell_id, cell)

        # Store the start and goal point IDs
        if cell == world_to_map(start_point):
            _start_id = cell_id
        if cell == world_to_map(goal_point):
            _goal_id = cell_id

        cell_id += 1

Connecting AStar points

Now that our Astar has a pool of points to work with, we need to connect them. This is so that it knows where it can move from any given point. We do this in the _connect_neighbor_cells() method.

This is why we’ve mapped our level design to tilemap coordinates so it’s easy to connect neighbor cells. We can use unit vectors to offset the current cell and get the neighbor cells around it. For instance, Vector2.RIGHT means one cell to the right.

For that, we turn the walkable_cell into an Array. At the moment it’s a PoolVector2Array, so it lacks some methods we need to operate on lists. With an Array, we have access to the find() method. We need this method to search the index of an element that corresponds to a value we pass to it. If we pass find(Vector2(12, 17), it will try to find a cell with that value and return its index. If that cell isn’t inside the list, it returns -1.

The AStarGrid doesn’t know what relative neighbors a cell should have. So let’s create a list of directions using a four-directional approach. Cells in any of these directions will be neighbor cells. We use the constants available in the Vector2 class to build our list of NEIGHBOR_DIRECTIONS. Note that you could add more directions to that list to make eight-directional path finding, for example.

const NEIGHBOR_DIRECTIONS := [
    Vector2.UP,
    Vector2.LEFT,
    Vector2.RIGHT,
    Vector2.DOWN
]

Now that we’ve defined the directions that build up neighbors, we can iterate on the _astar points using the get_points() method. We find the index of the point that matches a neighbor cell using the find() method. We’ve indexed our points, so their position on the list is also their ID.

With that in mind, we can use the neighbor_cell_id to connect neighbor cells to the current cell using the AStar2D connect_points() method. It asks for two points IDs, so we can pass the current cell ID and the neighbor_cell_id. To access the current cell ID we use the AStar2D get_point_position() method.

func _connect_neighbor_cells():
    # Turn the walkable_cells into an Array to find its elements indices
    var walkable_cells_array := Array(walkable_cells)

    # Iterate on each element of the get_points() list then look for neighbor
    # cells
    # offsetting the current cell using the directions in the
    # NEIGHBOR_DIRECTIONS list
    for point in _astar.get_points():
        # Figure out the current cell ID
        var cell = _astar.get_point_position(point)

        # Offset the current cell by one cell at each direction provided in the
        # NEIGHBOR_DIRECTIONS
        for direction in NEIGHBOR_DIRECTIONS:
            var neighbor_cell_id = walkable_cells_array.find(cell + direction)

            # Skip cells that weren't found
            if neighbor_cell_id == -1:
                continue
            # Connect the current cell to the current neighbor cell
            _astar.connect_points(point, neighbor_cell_id)

Creating a path with AStar

Now it’s time to ask the AStar2D to find a path between our points, this is our goal after all.

We create a _get_astar_path() method to return a PoolVector2Array containing cells that are part of the path. We store this path in a local variable called astar_path.

In this method, we need to make the necessary steps to build that path. First, we call the _create_astar_points() method. Then we connect the cells with the _connect_neighbor_cells() method. With that, we’ve fulfilled the requirements that allow the magic to happen.

We call the get_point_path() method on the _astar to get a path between the start_point and the goal_point. We pass the _start_id and the _goal_id as arguments and return the result.

func _get_astar_path() -> PoolVector2Array:
    var astar_path: PoolVector2Array

    _create_astar_points()
    _connect_neighbor_cells()

    # Creates a walkable path from the start_point to the goal_point
    astar_path = _astar.get_point_path(_start_id, _goal_id)
    return astar_path

Remapping the path to world coordinates

Remember that currently, we have map coordinates in the form of cells. We need to convert them to world coordinates so the Level can use them. The Level is going to use these points for the Wave.curve property so Enemies can walk on the path provided by the AStar. Isn’t that cool? How everything works together to solve ever-complex problems?

We create its first public method in AStarGrid. It’s through this method the Level will ask it to build a path that the enemies can walk on. We name it get_walkable_path().

This method should return a PoolVector2Array. We create a local variable called walkable_path which is initially empty.

We store the path provided by _get_astar_path() method and convert each point to world coordinates using the map_to_world() method.

Recall that cells have their origin at the top left corner. We need to offset the final coordinate because we want this path to fit in the center of the TileMap’s tiles. We can export a Vector2 variable called offset to use different offsets as we test the environment art. Since the default cell size is Vector2(64, 64), we use half this value to get the center of a cell by default.

After running through all cells and converting them to offset world coordinates, we can return the walkable_path.

export var offset := Vector2(32, 32)

func get_walkable_path() -> PoolVector2Array:
    var walkable_path := PoolVector2Array()

    var astar_path = _get_astar_path()

    # Convert the map cells in the `astar_path` to world positions
    for cell in astar_path:
        var point := map_to_world(cell)

        # Offset the positions to get the center of each cell instead of its top
        # left corner
        walkable_path.append(to_global(point) + offset)
    return walkable_path

AStarGrid.gd

# Calculates a set of walkable positions between the start_point and the
# goal_point
extends TileMap

const DIRECTIONS := [
    Vector2.UP,
    Vector2.LEFT,
    Vector2.RIGHT,
    Vector2.DOWN
]

export var start_point := Vector2.ZERO
export var goal_point := Vector2.ZERO
export var offset := Vector2(32, 32)

var walkable_cells: PoolVector2Array

var _astar := AStar2D.new()
var _start_id := 0
var _goal_id := 0


func get_walkable_path() -> PoolVector2Array:
    var walkable_path := PoolVector2Array()

    var astar_path = _get_astar_path()
    for cell in astar_path:
        var point := map_to_world(cell)
        walkable_path.append(to_global(point) + offset)
    return walkable_path


func _get_astar_path() -> PoolVector2Array:
    var astar_path: PoolVector2Array

    _create_astar_points()
    _connect_neighbor_cells()

    # Creates a walkable path from the start_point to the goal_point
    astar_path = _astar.get_point_path(_start_id, _goal_id)
    return astar_path


func _create_astar_points() -> void:

    var cell_id := 0
    for cell in walkable_cells:
        _astar.add_point(cell_id, cell)

        # Store the start and goal point IDs
        if cell == world_to_map(start_point):
            _start_id = cell_id
        if cell == world_to_map(goal_point):
            _goal_id = cell_id

        cell_id += 1


func _connect_neighbor_cells():
    # Turn the walkable_cells in an Array to find its elements indices
    var walkable_cells_array := Array(walkable_cells)

    for point in _astar.get_points():

Testing the PathFinding

Now it’s time to see if we’ve managed to make our path finding system. Back to the Level scene let’s add an instance of the AStarGrid as a child of the Level.

To make it explicit, add a Position2D node and name it StartPoint. Then add another and name it GoalPoint. Reparent the PlayerBase as a child of the Goal_point.

Let’s add a Sprite as child of the StartPoint. This will provide a visual cue of where the Enemies are going to appear. Drag the res://sprite-sheet.svg file to the Texture property. Then toggle the Region > Enabled property. From the Texture Region editor, enable Auto Slice and select the orange circle with a spiral.

Reset the PlayerBase to 0.0 on both axes. From now on, we only move the StartPoint and the GoalPoint to design our levels. To ensure that we won’t move the wrong nodes, let’s select all other nodes and lock them. For that, we can toggle the lock icon at the 2D editor tool shelf.

The AStarGrid is the last piece of the Level puzzle. We can finally start gluing everything together. Let’s attach a script to the Level node itself, we can save it as res://Levels/Level.gd.

For our test, we use the PathPreview to draw the path we receive through the AStarGrid. Create a reference to both nodes and the TileMap using onready variables. We use the TileMap to get which cells are available to walk on using the get_used_cells_by_id() method.

Let’s also store a reference to the StartPoint and GoalPoint nodes. This is because the Level must provide the start and goal points to the AStarGrid.

extends Node2D

onready var _path_preview := $PathPreview
onready var _astar_grid := $AStarGrid

onready var _tilemap := $TileMap
onready var _start_point := $StartPoint
onready var _goal_point := $GoalPoint

We erase the current points in the _path_preview in the _ready() callback. Then we inject the _start_point and the _goal_point’s position into the _astar_grid.

func _ready() -> void:
    _path_preview.clear_points()
    _astar_grid.start_point = _start_point.global_position
    _astar_grid.goal_point = _goal_point.global_position

Next up, we need to get the cells on the _tilemap that represent walkable cells. We need to pass them to the _astar_grid.walkable_cells property using the _tilemap.get_used_cells_by_id() method. It asks for the ID of the tile used in these cells. In our case, we can pass 2 as argument. This represents the Enemy Walking Path Cell tile in our tileset. This might vary depending on how you’ve set up your tileset. Instead of passing the ID directly, let’s create a constant that represents this ID called WALKABLE_CELLS_ID.

With that, we have everything set to calculate the walking path. We can use the _astar_grid.get_walkable_path() to also build the _path_preview.points. The final test script will look like this:

Level.gd

extends Node2D

const WALKABLE_CELLS_ID := 2

onready var _path_preview := $PathPreview
onready var _astar_grid := $AStarGrid

onready var _tilemap := $TileMap
onready var _start_point := $StartPoint
onready var _goal_point := $GoalPoint


# Delete this block after the tests.
# Begin of test lines
func _ready() -> void:
    _path_preview.clear_points()
    _astar_grid.start_point = _start_point.global_position
    _astar_grid.goal_point = _goal_point.global_position

    _astar_grid.walkable_cells = _tilemap.get_used_cells_by_id(WALKABLE_CELLS_ID)
    _path_preview.points = _astar_grid.get_walkable_path()
# End of test lines

With that, our PathPreview should now link the StartPoint and the GoalPoint. As long as they sit on a cell painted with the Enemy Walking Path Cell tile, we will be able to find a path between them!
