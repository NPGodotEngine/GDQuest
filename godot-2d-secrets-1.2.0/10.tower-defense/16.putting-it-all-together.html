Putting It All Together

At the minute, we only have mechanisms that behave and react to each other. We don’t have winning or losing conditions and we don’t reward players for defending their base. Players also can’t invest any resources to acquire new Towers.

In this lesson, we are going to set this up. We’ll create the actual representation of the Player in the game, with resources to manage. We’ll also provide the interface necessary for the player to interact with the game’s world. In doing this, you’ll learn to:

  • Manage in-game resources
  • Make a shop for tower purchase
  • Design an interface to trigger game states
  • Handle the game main loop with win and lose conditions

It’s time to assemble the game world we’ve created and allow players to interact with it. We’ll provide incentives to tell players if their strategies work and a goal to accomplish of course.

We’ll create an object called Game to put the toys we’ve been creating into context. This diagram shows how the Game uses them.

Managing in game resources

If you look to most strategy games, there are resources that players use to acquire new units in the game. For instance, in Warcraft III we have gold, food, and lumber. In our game, we scope it down to only one resource: gold.

Enemies reward players with an amount of gold when killed. To acquire new Towers, players must spend gold. If players decide to sell a Tower, they get some of their investment back. This is the design of our game’s inner economy.

The player object

To hold and manage the gold the player earns, we represent the player as a class in our game. This will be our abstraction of the player. Create a new scene and add a Node as root. Rename it as Player. Save the scene as res://Actors/Player/Player.tscn.

Every time the gold value changes, we emit a signal to update the interface.

Attach a new script to the Player node with the following code.

Player.gd

class_name Player
extends Node

signal gold_changed(gold_amount)

# We use a setter to ensure the value never goes below `0` and emit the
# `gold_changed` signal.
export var gold := 100 setget set_gold


func set_gold(new_amount: int) -> void:
    gold = max(0, new_amount)
    emit_signal("gold_changed", gold)

Buying new towers

Now we’re going to allow players to spend their cold hard gold and make new acquisitions!

Let’s create a shopping interface that displays available Towers for purchase. The TowerShop will tell the TowerPlacer what tower to place on the map. The shop handles the transaction, and the placer handles the delivery.

Displaying tower options

To display the Towers available for purchase, we create a button that holds a reference to the Tower PackedScene.

Let’s create a new scene and add a TextureButton as root. Rename it as UITowerPurchaseButton. Save the scene as res://UI/UITowerShop/UITowerPurchaseButton.tscn.

Then, add a Label as its child. The Label will display the Tower’s cost to let players know how much they spend purchasing it. Reposition the label below the button.

In the start project, there are some images under the res://UI/UITowerShop/ folder we can use for the button. This is to provide a visual cue for players when they hover or click on the button. We’ve named each image based on their respective Texture slot on the TextureButton. Drag the res://UI/UITowerShop/tower-icon-hover.svg to the Texture > Hover property and so on.

Change the Stretch Mode property to Keep Aspect Centered and toggle the Expand flag on. This allows the texture to expand and maintain its aspect ratio should we scale the button.

Change the button Action Mode property to Button Press. The button will consider itself pressed as soon as we press it, increasing responsiveness. The default Button Release only triggers the button when we release the click.

Note that for some buttons it might be a good thing to have their Action Mode set to Button Release.

Toggle on the Expand flag on both Horizontal and Vertical axes under the Size Flags category to allow the button to take all the available space when we add this button as a child of a container Control. It’s a good trick to distribute UI elements evenly.

Attach a script to the button and save it together with the scene as UITowerPurchaseButton.gd.

extends TextureButton

export var tower_scene: PackedScene

onready var _label := $Label

# Public as other classes access it
var tower_cost := 0

# Instance the Tower to access its data, e.g. cost and description
func setup_tower_data() -> void:
    var tower: Tower = tower_scene.instance()
    # Store the tower's cost and display the information
    tower_cost = tower.cost
    _label.text = "Cost: %s" % tower_cost
    # We've extracted all the data we need so we queue it to free up memory
    tower.queue_free()

When the player presses the button, it needs to notify other classes and passes the Tower PackedScene so they can handle the tower purchasing moving forward.

signal tower_purchased(tower_scene)

func _pressed() -> void:
    emit_signal("tower_purchased", tower_scene)

As soon as the UITowerPurchaseButton is ready, we should set it up. The script should look like this:

UITowerPurchaseButton.gd

extends TextureButton

signal tower_purchased(tower_scene)

export var tower_scene: PackedScene

onready var _label := $Label

var tower_cost = 0


func _ready() -> void:
    setup_tower_data()


# Instance the Tower to access its data, e.g. cost and description
func setup_tower_data() -> void:
    var tower: Tower = tower_scene.instance()
    tower_cost = tower.cost
    _label.text = "Cost: %s" % tower_cost
    tower.queue_free()


func _pressed() -> void:
    emit_signal("tower_purchased", tower_scene)

To test our UITowerPurchaseButton, drag the res://Objects/Towers/Tower.tscn file to the UITowerPurchaseButton > Tower Scene property.

In the UITowerPurchaseButton add a breakpoint at the line where we emit the tower_purchased signal to test the button is running the logic when pressed.

Drag the button to the middle of the screen and hit the Play Scene button. If everything goes as planned, the button should display the tower’s cost.

Click on the button and Godot should prompt its script right at the line where we added the breakpoint. Remove the breakpoint and we’re done!
