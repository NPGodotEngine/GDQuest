Designing the User Interface

It’s time put everything together. Create a new scene with a Node as root. Rename it as Game and save it as res://Game.tscn.

We’ll begin with the play screen interface. Note we have the res://UI/ folder with some interface elements in the start project. We’ve prepared them upfront to save you time.

The first thing to do is add a new CanvasLayer for the interface and name it UILayer. This will allow us to always render the UI elements on top of other elements. Add Level instance as a child of the Game as well.

Add a Control node as a child of the UILayer to allow us to distribute elements using anchors. It’s the parent of all UI elements, so let’s call it UI. Change its layout to Full Rect to spread its rect through the whole screen.

We can add another Control as a child of UI. This is our Heads Up Display, or HUD for short. It holds the elements that the player can interact with and read data from. Let’s name it HUD and change its layout to Full Rect as well.

As children of the HUD, instance the UITowerShop and the UIGoldPanel. You can find the gold panel in the res://UI/UIGoldPanel folder.

Add a Button and call it StartButton to allow players to “finish” their turn. As its Text property, we can say something like Start!.

When players lose, they should be able to try again. Add another Button and name it RetryButton. As its Text property, you guessed it, we display a Retry text.

Change the layout of this button to Center to make it clear that players can retry if they lose. This button should only appear when players lose, so hide it for now. After positioning these elements, we should end up with something like this:

Note that I’ve toggled the Level Visibility so we could see only the interface elements.

In our design, players aren’t able to interact with the game elements during the Play Mode. They can place towers and check the player base health only when in the Plan Mode. To prevent these interactions, we create a mouse barrier. It’s a Control node that consumes mouse events and prevents these events from reaching other nodes.

As a child of the UI node, add a new Control and name it MouseBarrier. Ensure it spams through the whole screen by setting its layout to Full Rect as well.

Make sure Mouse > Filter is set to Stop so it consumes any mouse events. This option only works when Controls are visible. Because our game begins in the Plan Mode, hide the MouseBarrier.

Note that, by default, all Control nodes behave like our MouseBarrier. That means that the UI and the HUD will also consume mouse events. But they shouldn’t, because players will interact with the game while they’re visible. Select other control nodes apart from the buttons and set their Mouse > Filter property to Ignore.

In the res://UI/UIScreenOverlay/ folder we have the UIScreenOverlay scene to make the transitions between the plan and the play modes. It has three methods to play animations according to the game’s state. The play_win(), play_lose(), and the play_wave_start(). They are all asynchronous methods which means their execution doesn’t finish as soon as we call them. Instead, these methods execute when the animation they call finishes.

We use the play_wave_start() method when transiting from the Plan to the Play mode.

Add an instance of UIScreenOverlay as a child of UI so we can use it later.

For now, the interface is only a visual aspect of the game. It has no impact on the game itself. But with the interface and the level in place, we can mediate their interactions using the Game node.
